#1
setInterval fires init() ~60× — all document event listeners stack on every tick
vB.js · lines 1047–1052 & 884–976
Critical Confirmed
Root Cause The bootstrap runs a setInterval at 50 ms for 3 s (~60 ticks). Each tick calls waitForElement("body", init, 50, 15000). Because body is always present, waitForElement fires init() on its very first inner tick, clears its inner interval, and returns. The outer 50 ms interval continues — spawning a new waitForElement every 50 ms, each immediately firing init() again. There is no re-entry guard in init().
Impact Every document.addEventListener inside init() — input (line 884), focus (line 896), blur (line 903), click via live() (line 922, 934), submit (line 976) — is registered up to 60 times. One Enter keypress in the ZIP field fires ~60 concurrent redirect attempts via window.location.href. Every blur shows the error up to 60 times; every click potentially triggers 60 analytics events. 
1047var testsignals = setInterval(function () {
1048    waitForElement("body", init, 50, 15000);  // body always present → init() fires every tick
1049}, 50);
1050setTimeout(function () {
1051    clearInterval(testsignals);  // stops after 3 s, but init() already ran ~60×
1052}, 3000);

// Inside init() — no guard, registered once per outer tick:
 884document.addEventListener('input',  function(e) { … });        // stacks 60×
 976document.addEventListener('submit', function(e) { … });       // stacks 60×
Fix
Add a one-time guard at the top of init(): if (window.TT_INIT_DONE) return; window.TT_INIT_DONE = true;. This lets the setInterval fallback tick safely while guaranteeing listener registration happens exactly once. Alternatively, call clearInterval(testsignals) from inside init() on its first successful run.
#2
Synchronous querySelector('.global-footer') inside composable-header callback — null on slow SPA silently skips all section injection
vB.js · line 829
Critical Confirmed
Root Cause Inside the waitForElement('div[class*="composable-customer-header"]', …) callback, the banner is injected first, then document.querySelector('.global-footer') is called immediately (line 829) — no polling, no wait. On a Next.js SPA, .global-footer is not guaranteed to be in the DOM at the same moment as the site header. The separate waitForElement('.global-footer', …) at line 874 only handles the custom footer nav links — it does not retry section injection.
Impact When bannerEl is null, the if (bannerEl) block at lines 831–870 is skipped entirely. The Related Services carousel, FAQ accordion, Reviews swiper, and Why-Hire section are never injected. The page renders with just the hero banner and no body content — a silent, complete failure with no console warning. 
 824waitForElement('div[class*="composable-customer-header"]', function() {
 825    if (!document.querySelector('.Cre_banner')) {
 826        // banner injected …
 829        var bannerEl = document.querySelector('.global-footer'); // ← no wait, may be null
 831        if (bannerEl) {   // ← ALL section injections gated here
 832            // services, FAQ, reviews, why_hire …
 870        }  // ← silently skipped when .global-footer not yet in DOM
 871    }
 872}, 100, 15000);
Fix
Move all four section injections (services, FAQ, reviews, why-hire) inside a waitForElement('.global-footer', function() { … }, 100, 15000) callback. This gives the SPA time to render the footer before the injection runs, regardless of which element (header vs. footer) appears first.
#3
loadSwiper has no in-flight flag — two identical CDN <script> tags injected when called twice in one init()
vB.js · lines 613–630, 837, 858
High Confirmed
Root Cause loadSwiper guards only against if (window.Swiper). Both calls at line 837 (services) and line 858 (reviews) run synchronously in the same init() execution. At line 858, the first <script> tag has been appended but its onload has not fired — window.Swiper is still undefined. Both calls fall through to script injection, appending two identical Swiper CDN tags.
Impact Two CDN scripts race to define window.Swiper. Both onload callbacks fire, calling initRelatedServicesSwiper() and initReviewsSwiper() twice each against the same DOM nodes. This produces duplicate Swiper instances, broken arrow navigation, doubled pagination bullets, and unpredictable slide counts. 
 613function loadSwiper(callback) {
 614    if (window.Swiper) { callback(); return; }  // ← only guard, no in-flight check
 626    var js = document.createElement('script');
 627    js.src = '…/swiper-bundle.min.js';
 628    js.onload = callback;
 629    document.head.appendChild(js);  // ← appended twice in the same init() run
 630}

 837loadSwiper(function() { initRelatedServicesSwiper(); }); // call 1
 858loadSwiper(function() { initReviewsSwiper(); });         // call 2 — window.Swiper still undefined
Fix
Add a pending-callback queue: var _swiperCbs = []; var _swiperLoading = false;. In loadSwiper: push the callback to _swiperCbs; if !_swiperLoading, set _swiperLoading = true, inject the single script, and on onload drain the queue by calling each callback. This ensures exactly one script tag and correct sequential initialization.
#4
buildInstantResultsUrl always appends &encoded_answers= with empty value
vB.js · lines 799–803
Medium Confirmed
Root Cause The URL template appends the literal string '&encoded_answers=' on line 802 with no value. The ternary on line 803 conditionally appends &project_pk=… but never fills in encoded_answers. The parameter name exists; its value is permanently empty on every redirect.
Impact Every CTA redirect URL carries &encoded_answers=. If the Thumbtack instant-results endpoint interprets a blank pre-answer as a submitted empty filter, search results may be silently miscategorized or filtered. At minimum, URLs diverge unexpectedly from the control page's redirect pattern, affecting result-set comparability. 
 799return 'https://www.thumbtack.com/instant-results/?keyword_pk=' + keyword_pk +
 800    '&zip_code=' + (zip || '') +
 801    '&ir_referrer=FRONT_DOOR_SEARCH' +
 802    '&encoded_answers=' +     // ← value never assigned
 803    (project_pk ? '&project_pk=' + project_pk : '');
Fix
Either remove &encoded_answers= entirely if not needed, or populate it conditionally: (cta.encodedAnswers ? '&encoded_answers=' + encodeURIComponent(cta.encodedAnswers) : ''). Check which property name the Thumbtack cta object uses — it may be cta.encoded_answers or cta.encodedAnswers.
#5
ZIP validation checks length only — non-digit 5-char strings (e.g. "AAAAA") pass and trigger a redirect
vB.js · line 984 (submit handler) & line 949 (click handler)
Medium Confirmed
Root Cause Both handlers check if (zip.length < 5). The HTML attributes inputmode="numeric" and pattern="[0-9]*" do not prevent letter input on desktop: inputmode only changes the virtual keyboard on mobile; pattern only runs on native form validation, which is bypassed here by e.preventDefault() in both handlers.
Impact A desktop user who types any five non-digit characters and submits is redirected to /instant-results/?…&zip_code=AAAAA&…, which returns empty or error results instead of showing a validation message. The user sees a broken results page with no explanation. 
// submit handler (line 984) — same pattern in click handler (line 949)
 984if (zip.length < 5) {   // ← "AAAAA".length === 5 → passes validation
 985    // show error …
 994    return;
 995}
 999var url = buildInstantResultsUrl(zip); // ← called with "AAAAA"
1001window.location.href = url;
Fix
Replace zip.length < 5 with !/^\d{5}$/.test(zip) in both the click and submit handlers. Consider extracting to a shared helper to eliminate the current triplication: function isValidZip(z) { return /^\d{5}$/.test(z); }
#6

vB.js · lines 999–1004 & 969–972
Medium Confirmed
Root Cause buildInstantResultsUrl accesses window.__NEXT_DATA__.props.pageProps.frontDoorPage.heroSection.filterSubsection.cta with no optional chaining or null guards. If any step is undefined — SSR payload changes, test environment, page variant — a TypeError is thrown and caught silently with console.info. Both the click and submit handlers share this pattern.

In a production scenario where Thumbtack's SSR payload diverges from the expected shape, the CTA silently breaks for all users.

1002} catch (err) {
1003    console.info('TT: could not build URL', err); // ← silent, no user feedback

In the catch, show the #tt-zip-error element with a fallback message: "Something went wrong — please try again.". Also add optional chaining in buildInstantResultsUrl: var cta = window?.__NEXT_DATA__?.props?.pageProps?.frontDoorPage?.heroSection?.filterSubsection?.cta; if (!cta) throw new Error('cta missing');
#7

vB.js · line 976 vs. click handler line 934
Low Plausible
Root Cause The click handler (line 934) calls both e.preventDefault() and e.stopPropagation(). The submit handler (line 976) calls only e.preventDefault(). Both are registered on document. Any other document-level submit listener — Next.js analytics, third-party tags, future routing integration — will still receive this event.
Impact Harmless on the current page where no competing submit listener is known to exist. Becomes a risk if analytics scripts or a router integration adds a document-level submit handler — potentially firing a spurious analytics event or triggering a route change. Also creates a maintenance inconsistency with the click handler.

#8
max-height: 60px overridden by max-height: 120px four lines later in same .TT-review-text ruleset
vB.css · lines 287 & 295
Low Confirmed
Root Cause The .TT-review-text ruleset contains two max-height declarations. The first (60 px, line 287) is overridden by the second (120 px, line 295). The first declaration is dead code. The intended clamp is 3 lines × 20 px line-height = 60 px, set via -webkit-line-clamp: 3.
Impact In browsers that support -webkit-line-clamp, the clamp wins and max-height is irrelevant. In browsers that do not support it, the fallback cap is 120 px (6 lines) instead of the intended 60 px (3 lines), showing more review text than designed and breaking card height uniformity in the swiper.

Fix
Remove the first max-height: 60px declaration (line 287). Verify whether 60 px or 120 px is the intended non-clamp fallback — if Figma shows 3-line clamp, the fallback should be 60 px, so remove line 295 instead and keep line 287.
