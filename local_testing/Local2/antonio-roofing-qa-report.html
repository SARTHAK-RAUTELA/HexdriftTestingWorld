Critical Bug — ZIP Code Form
Empty ZIP + Enter key navigates away from the page
When the user clears the ZIP field and presses Enter, the browser performs a native form GET submission — overwriting the URL and losing TT_QA=true.
Fixed in vB.js
Root cause: form#uniqueId2 has no action attribute and no submit event handler. When a user presses Enter inside any text input in a <form>, the browser fires a submit event and navigates via GET to the current URL with form fields as query params. The existing click listener on [data-test="hero-filters-cta"] does not intercept this — it only catches mouse clicks on the button element.

What happens step by step on empty ZIP + Enter:

  1. User focuses zip input → data-tt-touched=1 is set
  2. User clears zip value
  3. User presses Enter → browser fires blur (capture) then submit
  4. Blur handler adds error + disable class — but this does not stop form submission
  5. No submit handler exists → browser navigates to /san-antonio-tx/roofing?zip_code=
  6. TT_QA=true param is stripped → variation no longer loads on the next page

Control page behaviour

The control (/tx/san-antonio/roofing) uses native React to manage the zip form — its onSubmit handler prevents default and runs client-side validation. The variation must replicate this via a DOM-level submit listener.

Before fix — missing handler:

// vB.js — NO submit listener existed anywhere
live('[data-test="hero-filters-cta"]', 'click', function(e) {
    // click-only — Enter key bypasses this entirely
        // ❌ pressing Enter in zip input skips this whole block
});

After fix — submit handler added to vB.js (line 976–1000):

/* ─── ZIP form: block Enter-key native submit ─── */
document.addEventListener('submit', function(e) {
    if (!e.target || e.target.id !== 'uniqueId2') return;
    e.preventDefault();                       // ✅ blocks native navigation

    var zipInput = document.querySelector('#uniqueId2 [autocomplete="postal-code"]');
    var zip = zipInput ? zipInput.value.trim() : '';
    var errorEl = document.getElementById('tt-zip-error');

    if (zip.length < 5) {
        if (!errorEl) {
            errorEl = document.createElement('div');
            errorEl.id = 'tt-zip-error';
            errorEl.style.cssText = 'color:#d9232d;font-size:13px;margin-top:6px;font-weight:500;';
            errorEl.textContent = 'Please enter a valid 5-digit ZIP code.';
            if (zipInput && zipInput.parentNode)
                zipInput.parentNode.insertAdjacentElement('afterend', errorEl);
        }
        return;                                  // ✅ shows error, stays on page
    }

    if (errorEl) errorEl.remove();
    try {
        var url = buildInstantResultsUrl(zip);
        window.location.href = url;              // ✅ only redirects on valid 5-digit ZIP
    } catch (err) { console.info('TT: could not build URL', err); }
});

Warning #2 — swiper-button-next.swiper-button-disabled selector is too broad
Warning — CSS

In vB.css the rule .swiper-button-next.swiper-button-disabled { display: none; } matches all disabled next-buttons on the page, not just the variation's. If the native Thumbtack page has any other Swiper instances, this could hide their next button unexpectedly.

Fix: scope it: .TT-swiper-btn.swiper-button-next.swiper-button-disabled { display: none; }

Current (risky) .swiper-button-next.swiper-button-disabled { display: none; }
Recommended (scoped) .TT-swiper-btn.swiper-button-next.swiper-button-disabled { display: none; }

The .swiper-button-prev version correctly already requires .TT-swiper-btn. Make the next button rule consistent with it.

CSS Issues

Warning — CSS

The following rule blocks appear twice in vB.css. Not a functional bug (last declaration wins and they're identical), but increases file weight and makes maintenance harder:

  • .TT-services-swiper
  • .TT-services-swiper .swiper-slide
  • .TT-service-card
  • .TT-service-card:hover

Leading-space class selector — intentional but fragile
[class*=" hero-header_heroContainer"] img { height: 232px; }
Note

The space before hero-header_heroContainer in the attribute selector means it only matches when that class is NOT first in the class attribute. This is a known technique to require a preceding class (acting as a namespace guard). It works as long as Thumbtack's React doesn't reorder the class list. Monitor after any framework upgrades.
