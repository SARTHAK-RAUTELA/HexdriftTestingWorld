2 Bugs Found & Fixed
All 12 bugs below were identified through code-level analysis comparing vB.js / vB.css against the reference clone DOM (Round 1: 7 bugs; Round 2 deep UI audit: 5 additional bugs). All have been fixed in the variation files before this report was published.
BUG-01 Broken review avatar images — leading whitespace in URL for Marcus T. and Linda R. Critical Fixed

File: vB.js — sectionData.reviews array, lines 85–86

Problem: Two avatar src strings had a leading space character before https://, resulting in an invalid URL that causes both images to return a 404/network error. The browser cannot resolve " https://…" as a URL.

Affected reviewers: Marcus T. (J. Stevens Roofing Co.) and Linda R. (Legacy Home Improvements)

Before: " https://fastly.picsum.photos/id/458/48/48.jpg?…"

After: "https://fastly.picsum.photos/id/458/48/48.jpg?…"

Removed leading space from both avatar URL strings.
BUG-02 overflow: hidden on #__next blocks vertical scrolling on mobile High Fixed

File: vB.css — line 690

Problem: The rule body.Antonio_Roofing div#__next { overflow: hidden !important; } was intended to suppress the horizontal overflow created by the "Why Thumbtack" section's ::before/::after pseudo-elements (which extend the gray background to full width). Using overflow: hidden (both axes) instead of overflow-x: hidden risks clipping vertically overflowing content and can trap scroll inside #__next on iOS Safari, making the page unscrollable past fold.

Before: overflow: hidden !important

After: overflow-x: hidden !important

Changed to overflow-x: hidden — horizontal overflow (from pseudo-elements) is still clipped; vertical scroll is unaffected.
BUG-03 Service pills have no horizontal gap — all chips visually merge into one block High Fixed

File: vB.css — .tt-cr-service-pill rule

Problem: The pills are display: inline-flex and the JS builds them with .join("") (no whitespace between). Without a margin-right, sibling pills directly touch each other, making "Gutter Cleaning" look joined to "Siding", etc. The row still wraps but rows above/below have no horizontal spacing at all.

Reference behavior: Clone shows pills separated with clear visual breathing room.

Added margin-right: 8px to .tt-cr-service-pill.
BUG-04 Trust-badge card constrained to 300 px on mobile — breaks layout on narrow viewports Medium Fixed

File: vB.css — .roofing-card and its mobile override

Problem: The base rule sets max-width: 300px on .roofing-card. The mobile media query (≤768 px) correctly sets width: 100%, but the max-width cap is never cleared, so on a 375 px phone the card is still limited to 300 px — leaving an awkward right gap instead of stretching full-width under the text block.

Added max-width: 100% inside the @media (max-width: 768px) block for .roofing-card.
BUG-05 Double border between adjacent .tt-cr-section elements Medium Fixed

File: vB.css — .tt-cr-section rule

Problem: .tt-cr-section had both border-top and border-bottom. When the Pricing section (which has no top border) sits between the Related Services section and the Neighborhoods section, the Neighborhoods section's border-top collides with Pricing's border-bottom, producing a 2 px double-line separator. Same issue between Neighborhoods → Reviews.

Removed border-top from .tt-cr-section. Added border-top: 1px solid #e9eced to .tt-cr-injected wrapper so the block still has a clean top separator against the Thumbtack page content above.
BUG-06 Invalid height attribute on hero <img> — uses CSS unit inside HTML attribute Low Fixed

File: vB.js — bannersection template literal, hero <img> tag

Problem: The img had height="500px". HTML dimension attributes must be unitless integers; CSS units belong in a style attribute. While most browsers silently accept this, it fails HTML validation and may cause unexpected layout in strict rendering contexts.

Changed height="500px" → height="500" (unitless integer per HTML spec).
BUG-07 Duplicate CSS rule blocks for .TT-services-swiper and .TT-service-card Low Fixed

File: vB.css — approx. lines 103–215

Problem: The rules for .TT-services-swiper, .TT-services-swiper .swiper-slide, .TT-service-card, and .TT-service-card:hover appeared twice in the CSS file. The second block was an exact copy of the first, causing unnecessary file size increase and potential cascade confusion if the blocks ever diverge.

Removed the second (duplicate) block, retaining the first occurrence only.
BUG-08 Hero subtext paragraph completely missing from variation (mobile overlay and desktop panel) High Fixed

File: vB.js — bannersection template literal

Problem: The reference clone prominently shows a two-sentence subtext under the hero headline: "San Antonio pros ready for hail damage, storm repair, and full replacements. Get matched with verified local roofers — not out-of-town storm chasers." This text was completely absent from the injected bannersection HTML in the variation — both in the mobile overlay (hero_header_parent_mobile) and in the desktop filter panel. Only the "Confirm your location to see quality pros near you." helper text was present.

Reference: Subtext appears in white directly below the headline on mobile, and in dark text above the zip field on desktop.

Added the full subtext paragraph to the mobile overlay (<p class="tt-hero-sub mb3">) and to the desktop filter panel (<p class="Type_text1__634gq mb2 tt-hero-sub-desktop">), positioned above the existing "Confirm your location" helper paragraph.
BUG-09 Hero CTA button text reads "Find a pro" — reference says "Find me a pro" High Fixed

File: vB.js — bannersection template literal, CTA button span

Problem: The primary CTA button in the hero panel rendered "Find a pro". The reference clone clearly shows "Find me a pro". The extra word "me" is intentional in the reference — it personalises the CTA and is part of the tested variant copy.

Changed button span text from "Find a pro" → "Find me a pro".
BUG-10 Marcus T. review shows 4 stars — reference shows 5 stars for all reviewers Medium Fixed

File: vB.js — sectionData.reviews, Marcus T. entry

Problem: The reference clone HTML renders five filled gold star SVGs for every review card (Ashley B., Marcus T., Linda R., David K.). The variation data had stars: 4 for Marcus T., producing four filled stars and one empty star — visually inconsistent with the reference and with the other three reviewers in the variation.

Changed Marcus T.'s stars value from 4 → 5 to match the reference.
BUG-11 Review stars use Unicode characters (★/☆) instead of SVG elements — color/size inconsistent across browsers Medium Fixed

File: vB.js — buildStars() function

Problem: buildStars() rendered Unicode ★ / ☆ characters inside spans. The reference clone uses proper SVG star elements with an explicit fill="#f5a623" attribute. Unicode star rendering differs across OS/browser combinations — size, weight, vertical alignment, and exact gold tone all vary — making the review section inconsistent across Chrome/Edge/Firefox/Safari/WebKit.

Replaced the Unicode approach with two SVG string constants (filledStar / emptyStar) using the same 5-point star path as the reference, with fill="#f5a623" for filled and fill="#ddd" for empty. Updated .tt-cr-star CSS from color/font-size to display:inline-flex.
BUG-12 Service pill arrow is plain text character "›" instead of SVG chevron — also missing CSS for hero subtext classes Medium Fixed

File: vB.js — buildRelatedServices(); vB.css — missing .tt-hero-sub / .tt-hero-sub-desktop rules

Arrow issue: The reference clone chips use a proper SVG right-pointing chevron path (M6.764 14.646L13 9…) inside each pill. The variation used a plain › HTML entity, which renders differently in weight, size, and vertical alignment across browsers and OS.

CSS gap: The hero subtext paragraphs added in BUG-08 were injected with classes tt-hero-sub and tt-hero-sub-desktop that had no corresponding CSS rules. On mobile, the text would inherit the white parent but had no explicit size/opacity. On desktop, it would pick up only Thumbtack's Type_text1__634gq base style with no distinction from the "Confirm your location" helper text.

Arrow: replaced <span class="tt-cr-arrow">›</span> with an inline SVG (class="tt-cr-pill-chevron") using the same chevron path as the reference; updated CSS from .tt-cr-arrow to .tt-cr-pill-chevron (16×16 inline-block SVG).
CSS: added .hero_header_parent_mobile .tt-hero-sub (white, 15px, 0.92 opacity) and .tt-hero-sub-desktop (#2f3033, 14px, 500 weight, 0.88 opacity) rules with a mobile hide for the desktop variant.
